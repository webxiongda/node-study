# Day 53 — 性能优化：实操 Demo

## Demo 1：用 autocannon 压测接口

### 实操目标
把本章核心能力落到一个最小可运行场景中，先完成闭环，再考虑封装和优化。

### 前置条件
- 已完成前面章节的基础项目结构。
- 本地能运行 Node.js/NestJS 或 Next.js 项目。
- 如果涉及数据库、Redis、队列或部署，先确认对应服务可连接。

### 操作步骤
1. 明确输入和输出：输入来自页面、接口、脚本或定时任务，输出必须可观察。
2. 新增最小实现：只写支撑 Demo 的必要代码，避免一次性扩展太多功能。
3. 加入日志或返回值：确保你能判断执行到了哪一步。
4. 制造一个失败场景：验证错误提示、日志或回滚是否合理。
5. 记录命令、结果和问题，后续写入项目 README。

### 示例代码或命令
```bash
# 按当前项目实际包管理器选择 pnpm/npm/yarn
pnpm install
pnpm dev

# 如果是后端接口 Demo，可用 curl 验证
curl http://localhost:3000/health
```

### 运行结果
- 功能可以通过命令、页面或接口稳定复现。
- 失败时能看到明确错误信息。
- 关键步骤有日志或可检查的数据变化。

### 常见错误
- 只写了代码，没有验证输入输出。
- 环境变量写死在代码里。
- 没有处理失败路径，导致面试追问时说不清。

## Demo 2：分析慢查询和 Node.js 性能瓶颈

### 实操目标
在 Demo 1 的基础上加入真实业务约束，让实现更接近简历项目。

### 前置条件
- Demo 1 已经跑通。
- 已知道相关模块的职责边界。
- 准备一组正常数据和一组异常数据。

### 操作步骤
1. 把 Demo 1 的实现接入现有业务模块。
2. 给关键输入补充校验或权限判断。
3. 增加异常场景：空数据、重复请求、权限不足、服务不可用。
4. 用日志、数据库记录或页面状态确认结果。
5. 总结这次改动可以如何写进项目亮点。

### 示例代码或命令
```bash
# 示例：运行测试或本地检查
pnpm test
pnpm lint

# 示例：查看接口响应
curl -i http://localhost:3000/api/example
```

### 运行结果
- 正常路径能完整执行。
- 异常路径不会让服务崩溃。
- 可以讲清楚“为什么这样设计”。

### 常见错误
- 没有考虑鉴权、幂等或并发。
- 数据库、缓存、队列状态不一致时没有排查路径。
- Demo 代码和业务代码混在一起，后续难维护。
