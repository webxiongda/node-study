- **Cookie、JWT、Refresh Token 的使用边界**：需要能说清它解决什么问题、适合什么场景、常见失败点是什么。
- **路由保护和权限守卫**：需要能说清它解决什么问题、适合什么场景、常见失败点是什么。
- **登录态过期后的用户体验**：需要能说清它解决什么问题、适合什么场景、常见失败点是什么。

## 使用场景
- 注册登录退出
- 刷新页面保持登录
- 后台页面鉴权
- 不同角色看到不同菜单

## 工作原理
本章的学习重点不是孤立记 API，而是把它放入 Node.js 全栈项目链路中理解：

```text
需求/流量/用户操作
  -> 接口或页面入口
  -> 业务模块处理
  -> 数据库/缓存/队列/外部服务
  -> 日志、监控、错误处理
  -> 用户可感知的结果
```

你需要能回答三个问题：

1. 这个能力放在链路的哪一层？
2. 它降低了什么成本或风险？
3. 如果它失效，系统会出现什么表现，应该如何兜底？

## 常见问题

| 问题 | 典型原因 | 处理思路 |
|------|----------|----------|
| 只会照着示例写 | 没有抽象出输入、输出、边界 | 先画流程，再写代码 |
| 本地能跑，联调失败 | 环境变量、网络、权限或数据状态不一致 | 固化配置示例和检查命令 |
| 面试表达空泛 | 没有项目场景支撑 | 用博客系统或全栈项目举例 |
| 忽略失败路径 | 只关注 happy path | 补充超时、重试、回滚、日志 |

## 底层逻辑
- 先区分“开发体验”和“生产可靠性”：前者追求快速反馈，后者关注稳定、可观测、可恢复。
- 先明确边界，再选择工具：工具只是实现方式，真正重要的是接口契约、数据流和失败处理。
- 面试回答要从业务出发：为什么需要、怎么落地、有哪些坑、如何验证。

## 与其他技术的对比

|--------|----------|----------|----------|
| 复杂度 | 中等，适合当前学习项目 | 更简单方案可能牺牲扩展性 | 看项目阶段和团队维护成本 |

## 面试高频问题
**Q：全栈项目-认证模块在项目里主要解决什么问题？**
A：围绕“注册登录退出”这类场景，把实现从能跑提升到可维护、可扩展、可排障。回答时要结合当前博客/全栈项目，而不是只背概念。

**Q：学习 全栈项目-认证模块 时最容易忽略什么？**
A：不要只记命令或 API，要说明边界条件、失败恢复和与前后章节的衔接。比如 前后端认证状态同步、Cookie、JWT、Refresh Token 的使用边界 的取舍要放进真实链路里解释。

**Q：面试官追问“为什么这样设计”时怎么答？**
A：先说业务约束，再说技术方案，最后说风险和替代方案。避免只回答“因为框架推荐”或“网上都是这么写”。
