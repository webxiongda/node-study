# Day 12 — 项目任务：博客系统 API 设计文档

## 业务背景

你要开始开发一个博客平台的后端，在写代码之前先设计好 API（API First 开发方式），作为前后端联调的接口契约。

## 任务

设计博客平台的完整 API，包含以下5个资源：**用户、文章、评论、标签、点赞**。

## 输出格式

用 Markdown 表格写出每个接口：

```markdown
## 用户 API

| 方法 | URL | 描述 | 认证 | 请求体 | 成功响应 |
|------|-----|------|------|--------|---------|
| GET | /api/v1/users/me | 获取当前用户信息 | 必须 | - | 200 User |
...
```

## 验收标准

- [ ] 每个资源至少有完整的 CRUD 操作（GET 列表、GET 单个、POST、PATCH、DELETE）
- [ ] URL 符合 REST 规范（名词复数、小写、连字符）
- [ ] 关联资源用嵌套 URL（如文章的评论）
- [ ] 明确标注哪些接口需要认证
- [ ] 每个接口有明确的成功响应状态码（200/201/204）
- [ ] 列表接口有分页参数说明

## 常见坑

1. **点赞**：`POST /posts/:id/likes`（点赞）和 `DELETE /posts/:id/likes`（取消点赞）是合理的设计——点赞本身是一个资源（没有 id，每用户只有一个）。
2. **搜索**：`GET /posts/search?q=xxx` 还是 `GET /posts?q=xxx`？推荐后者，搜索只是过滤，不需要单独的路径。
3. **文章发布状态**：`POST /posts/:id/publish` 还是 `PATCH /posts/:id` with `{ published: true }`？两种都可以，前者更明确。
