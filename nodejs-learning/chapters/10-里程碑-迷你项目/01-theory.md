# Day 10 — 里程碑：迷你项目 Review · 理论文档

## 核心概念：代码质量与项目交付

Day 10 不是学新知识，而是**整合 Day 01-09**，把学过的内容落地成一个可以展示给面试官的项目。

---

### 1. 项目结构规范

```
mini-todo-api/
├── package.json
├── README.md           ← 面试官最先看这里
├── src/
│   ├── index.js        ← 入口文件（启动服务）
│   ├── app.js          ← 应用组装（中间件 + 路由）
│   ├── router.js       ← 路由器实现
│   ├── middleware/
│   │   ├── logger.js
│   │   ├── cors.js
│   │   ├── bodyParser.js
│   │   └── errorHandler.js
│   ├── routes/
│   │   └── todos.js    ← TODO 路由
│   └── errors.js       ← 自定义错误类
└── test/
    └── todos.test.js   ← curl 命令测试脚本
```

**为什么要分层？** 单文件很快，但不可维护。分层的好处：
- 每个文件单一职责
- 中间件可复用
- 路由清晰可见
- 便于面试时讲解

---

### 2. README 写法（⭐ 面试重要）

好的 README 决定面试官第一印象：

```markdown
# Mini TODO API

用原生 Node.js 实现的 TODO REST API，无框架依赖。

## 技术亮点
- 手写路由器（正则匹配 + 参数提取）
- 中间件链（支持 async/await）
- 分层错误处理（自定义 Error 类 + 全局处理）
- 分页 + 搜索 + 过滤查询

## 快速开始
node src/index.js

## API 文档
| 方法 | 路径 | 描述 |
|------|------|------|
| GET | /todos | 获取列表（支持分页/搜索）|
...

## 设计决策
- 为什么不用 Express：目的是理解框架底层原理
- 内存存储：用于演示，生产应替换为数据库层
```

### 3. 代码质量检查清单

面试前/提交前必查：

- [ ] 函数单一职责（一个函数做一件事）
- [ ] 统一的错误处理（不直接 `console.error` 暴露给用户）
- [ ] 所有异步函数都有错误处理（try/catch 或 .catch）
- [ ] 输入验证（必填字段、类型检查）
- [ ] 边界情况处理（分页越界、空数组、无效 ID）
- [ ] 没有 `console.log` 调试代码残留
- [ ] 常量提取（魔法数字如 `1000000` → `MAX_BODY_SIZE`）

---

### 4. 如何向面试官展示这个项目

**推荐的讲解顺序：**
1. 说明这是**不用任何框架**实现的 REST API，目的是理解框架底层
2. 展示项目结构，说明每层的职责
3. 讲路由器实现（正则、参数提取）
4. 讲中间件链（next()、错误处理、洋葱模型）
5. Demo：用 curl 跑一遍 CRUD

**可能被问到的追问：**
- "如果改用 Express，哪些代码可以删除？"（路由器、中间件引擎、body parser → 全换成 express 内置的）
- "这个项目怎么加数据库？"（答：将内存数组替换为数据库查询层，接口不变）
- "如何测试这个 API？"（答：curl / Postman 手动测，后续学 supertest 自动化测试）
