# Day 33 — Redis 缓存策略：复习文档

## 核心知识点总结

### 三大问题一句话区分

| 问题 | 一句话 | 根因 |
|------|--------|------|
| 穿透 | 查鬼（key 根本不存在） | 恶意请求/脏数据 |
| 击穿 | 锁门瞬间（热点 key 过期） | 高并发 + 热点 |
| 雪崩 | 集体失效（大量 key 同时过期/Redis 宕机） | 统一 TTL/单点 |

### 解决方案速查

| 问题 | 方案 | 核心代码/命令 |
|------|------|-------------|
| 穿透 | 缓存空值 | `cache.set(key, 'NULL', 60s)` |
| 穿透 | 布隆过滤器 | 请求前过滤不存在的 ID |
| 击穿 | 互斥锁 | `SETNX lockKey 1 EX 10` |
| 击穿 | 逻辑过期 | value 内存 expireAt，异步重建 |
| 雪崩 | TTL 随机化 | `baseTTL + random() * 0.2 * baseTTL` |
| 雪崩 | 高可用 | Redis Sentinel / Cluster |

## 易错点整理

1. 互斥锁必须做**双重检查**：拿到锁后再查一次缓存，因为等待期间其他线程可能已经重建
2. 缓存空值时 TTL 要短（1分钟），否则真实数据恢复后还要等很久
3. 逻辑过期方案：返回的是旧数据，只适合对一致性要求不高的场景
4. 防雪崩的 TTL 抖动要在**每次写缓存时**加，不是启动时算好固定值

## 高频面试题

**Q1：缓存穿透、击穿、雪崩分别是什么？怎么解决？**  
穿透：查不存在的 key → 空值缓存/布隆过滤器  
击穿：热点 key 过期 → 互斥锁/逻辑过期  
雪崩：大量 key 集中过期 → TTL 随机化/Redis 高可用

**Q2：互斥锁方案为什么要双重检查？**  
第一个拿到锁的线程重建缓存后，等待的线程拿到锁时不需要再重建，直接读缓存即可。少一次 DB 查询。

**Q3：布隆过滤器的误判率是什么意思？**  
布隆过滤器可能把不存在的 key 误判为「可能存在」（假阳性），但不会把存在的 key 判断为不存在。所以会有少量穿透，但正常数据一定能通过。

**Q4：逻辑过期和物理过期的区别？**  
物理过期：key 到 TTL 会被 Redis 删除，重建期间所有请求排队或穿透。  
逻辑过期：key 永远存在，value 里有 expireAt 字段，过期时异步重建，返回旧数据，无阻塞。

**Q5：Write-through 和 Cache-aside 的区别？**  
Cache-aside：应用层控制，读缓存→DB，写 DB→删缓存；简单灵活，最常用。  
Write-through：写时同步更新缓存和 DB，强一致，但写操作较慢。
