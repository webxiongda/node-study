# Day 51 — 项目任务：补齐核心模块单元测试

## 业务背景
你正在完善一个面向面试展示的 Node.js 全栈项目。当前目标不是堆功能，而是让项目具备真实业务系统的关键能力：可运行、可解释、可排查、可迭代。

本章任务聚焦：**单元测试**。

## 技术要求
- 必须结合现有项目结构，不要写成孤立脚本。
- 代码要有明确的输入、输出和错误处理。
- 涉及配置时使用环境变量或示例配置，不要硬编码敏感信息。
- 涉及接口时保持 RESTful 命名和统一响应风格。
- 涉及前后端联调时，要处理加载态、空状态和错误态。

## 输入输出

### 输入
- 用户操作、HTTP 请求、定时任务或后台事件。
- 必要的参数、认证信息和业务数据。

### 输出
- 关键 Service 测试、边界用例、Mock 策略、覆盖率截图或说明。
- 一段可以写进 README 或简历的项目说明。

## 代码实现
你需要自己完成实现。建议按下面顺序推进：

1. 先写最小可运行版本。
2. 再接入真实业务模块。
3. 补充异常路径和日志。
4. 用命令、页面或测试验证。
5. 记录踩坑和改进点。

> 注意：这里不直接提供完整答案。你完成后可以把代码交给 AI 做 review。

## 验收标准
- 功能在本地可以稳定复现。
- 正常路径和至少 2 个异常路径都被验证。
- 关键配置不硬编码。
- README 或学习记录中能说明设计取舍。
- 你能用面试表达讲清楚“为什么这么做”。

## 常见坑
- 只完成 happy path，没有失败处理。
- 只实现功能，没有可观察的日志或结果。
- 代码和业务边界混乱，后续难维护。
- 不能解释方案取舍，被追问时只能背 API。
