# Day 33 — Redis 缓存策略：验收自测

## 题 1：概念辨析

描述以下三种情况属于哪类缓存问题，并说明原因：

A. 黑客用脚本随机生成不存在的用户 ID（如 -1, 0, 99999999）请求接口，导致数据库 CPU 飙升  
B. 热门商品的缓存在双11秒杀前5秒过期，10万用户同时请求商品详情  
C. 凌晨3点定时任务批量刷新所有文章缓存，同时删除了5万个 key，5分钟后全部重新生成时数据库崩溃

## 题 2：方案选择

对于以下场景，选择最合适的防护方案并说明理由：

1. 电商平台防止爬虫用随机 productId 攻击（productId 合法值可知）
2. 新闻网站头条文章（流量极高），但内容可接受5秒的轻微延迟

## 题 3：代码题

以下代码有什么问题？如何修复？

```ts
async getUser(id: number) {
  const cached = await this.cache.get(`user:${id}`);
  if (cached) return cached;
  
  const user = await this.db.findUser(id);
  await this.cache.set(`user:${id}`, user, 3600000); // 1小时
  return user;
}
```

## 题 4：业务场景设计

你在做一个博客平台，有以下需求：  
- 文章详情缓存5分钟  
- 平台有1000篇文章，每天零点统一刷新缓存  
- 最近有攻击者在批量请求不存在的文章 ID  

请设计完整的缓存策略，说明如何防穿透、防雪崩。

## 参考答案

**题 1**：  
A = **缓存穿透**（请求的 key 在数据库中根本不存在）  
B = **缓存击穿**（单个热点 key 在高并发时过期）  
C = **缓存雪崩**（大量 key 集中同时过期）

**题 2**：  
1. 布隆过滤器（productId 有限且可预知，可全量加入布隆过滤器，快速拦截非法 ID）  
2. 逻辑过期方案（不设物理 TTL，读到过期时异步重建缓存，本次返回旧数据，避免击穿且无感知延迟）

**题 3**：  
问题：当 `user` 为 `null`（用户不存在）时，仍然会缓存 `null`，下次 `if (cached)` 判断 `null` 为假，每次都穿透 DB（防穿透失效）。  
修复：检查用户是否存在，若不存在则缓存特殊标记（如 `'NULL'`），读时特殊处理。

**题 4**：  
- 防穿透：缓存空值（不存在的文章 ID 写入 `'NULL'`，TTL 1分钟）  
- 防雪崩：零点刷新时给每篇文章 TTL 加 ±30秒随机抖动，分散重建时间  
- 额外：限流（ThrottlerGuard），防止攻击者大量请求
